# Commit Message Rules

This repo follows [Conventional Commits](https://www.conventionalcommits.org/). Every commit message must match:

```
<type>(<scope>): <subject>
```

The scope is optional; the colon and space after the type (or scope) are required.

These rules are not just a style guide — releases are generated by [`standard-version`](https://github.com/conventional-changelog/standard-version), which parses commit messages to decide the next version number and to populate `CHANGELOG.md`. Bad commit messages produce wrong release notes.

## Allowed types

This repo follows the [Angular commit type convention](https://github.com/angular/angular/blob/main/CONTRIBUTING.md#type) that the Conventional Commits spec recommends. Use one of:

| Type       | Use for                                                                                  |
| ---------- | ---------------------------------------------------------------------------------------- |
| `feat`     | A new user-visible feature or setting (triggers a **minor** version bump)                |
| `fix`      | A bug fix (triggers a **patch** version bump)                                            |
| `perf`     | A change that improves performance without changing behavior                             |
| `refactor` | A code change that neither fixes a bug nor adds a feature                                |
| `style`    | Formatting / whitespace changes that do not affect meaning (Prettier, ESLint --fix)      |
| `test`     | Adding or correcting tests                                                               |
| `docs`     | Documentation only (`README.md`, `AGENTS.md`, `BUILD.md`, `CONTRIBUTING.md`, code docs)  |
| `build`    | Build system, bundler config, package manager, lockfile, dependencies (`esbuild.config.mjs`, `package.json` scripts/deps, `pnpm-lock.yaml`) |
| `ci`       | GitHub Actions workflows under `.github/workflows/`                                      |
| `chore`    | Maintenance work that does not fit the categories above                                  |

If a change touches multiple types, pick the one that best describes the *primary* purpose of the commit. If two types feel equally important, that is usually a sign the change should be split into two commits.

## Allowed scopes

Scopes are optional. Real scopes from this repo's history that you can reuse when relevant:

- `outline` — code in `src/outline/` and outline-view integration (e.g. `feat(outline): add bidirectional sync to outline view`)
- `settings` — settings UI / persisted settings (e.g. `feat(settings): Add option to always fold creases when opening a file`)
- `foldmarker`, `legacyeditor` — specific subsystems referenced in past `fix:` commits
- `release` — automatically attached by `standard-version` to release commits (do not write these by hand)

Otherwise, prefer no scope unless adding one genuinely clarifies the change. Do not invent new scopes ad-hoc.

## Subject rules

- Use the imperative mood ("Add command", not "Added command" or "Adds command").
- Capitalize the first word of the subject. (Existing history is mixed; prefer capitalized going forward.)
- No trailing period.
- Keep the subject under ~72 characters. If you need more detail, use the body.
- Reference issues/PRs in the subject only when the change is "this PR's followup". Otherwise let GitHub link them.

## Body (optional)

Use a body when the *what* in the subject does not explain the *why*, or when the change has multiple notable points.

- Separate the body from the subject with a blank line.
- Wrap at ~72 characters.
- Bullet lists are fine; the existing history uses both `*` and `-` bullets.

## Breaking changes

Breaking changes are signalled with a `BREAKING CHANGE:` footer in the commit body. `standard-version` reads this footer to:

1. Cut a **major** version bump (e.g. `0.7.0` → `1.0.0`).
2. Highlight the change in `CHANGELOG.md` under a **⚠ BREAKING CHANGES** heading.

Real example from version `0.7.0` (see [`CHANGELOG.md`](CHANGELOG.md)):

```
fix: no longer causes a split view to appear when opening a file.

BREAKING CHANGE: removed the options to sync folds between the editor
and the Outline view. Because of the internal Obsidian changes, this
logic has been broken for a while and has been removed.
```

The `!` shorthand (`feat!: drop X`) also works, but the footer form is what this repo's history uses.

## Releases via standard-version

**Do not write `chore(release):` commits by hand.** They are generated by:

```bash
pnpm run release
```

`standard-version` walks the commits since the last tag, decides the next version (patch / minor / major) based on the types it sees, updates `manifest.json` + `package.json` + `CHANGELOG.md`, makes a single `chore(release): X.Y.Z` commit, and tags it. The tag is unprefixed (`0.7.0`, not `v0.7.0`) per the `"standard-version": { "t": "" }` config in `package.json`.

This means:

- Hand-written `chore(release):` commits will collide with the next `pnpm run release` run.
- The `release` scope on any commit other than a release commit confuses the changelog.
- If you need to override the version (e.g. force a major bump), pass it explicitly: `pnpm run release -- --release-as major`.

See [`CONTRIBUTING.md`](CONTRIBUTING.md) for the full release workflow.

## Examples

Good (drawn from this repo's history and `standard-version`'s expectations):

- `feat: Add commands to increase fold level for file`
- `feat(outline): add bidirectional sync to outline view`
- `feat(settings): Add option to always fold creases when opening a file`
- `fix: take blockId into consideration when calling 'toggleCrease'`
- `fix(foldmarker): Make sure the fold marker only shows in a properly escaped comment`
- `build: use node v18`
- `build: skip eslint when building for release`
- `docs: Document the release process`
- `chore: fix changelog urls` *(maintenance not generating a release entry)*

Avoid:

- `Cleanup`, `tmp`, `Update package.json` — not parseable by `standard-version`, will be silently skipped from the changelog.
- Hand-writing `chore(release): X.Y.Z` — let `standard-version` produce these.
- `Update README` — be specific: `docs: Document install instructions`.

# TypeScript Style

## Don't sprinkle `as const`

Avoid `as const` on string literals (especially discriminants like `type: "dropdown"` or property keys) inside object literals that are passed to a typed API. When the receiving function's parameter type is a discriminated union or has literal-typed fields, TypeScript already narrows the literal — the cast is noise.

Only reach for `as const` when:

- the value is being assigned to a `let` / variable whose inferred type would otherwise widen (e.g. `const mode = "edit"` is fine; `let mode = "edit"` widens to `string`), **and** that widening is actually breaking something.
- you are building a tuple or readonly array whose literal types need to be preserved across a function boundary.

If a real type error appears, fix the type, not the call site.
